# OrchestratorR1: 通过强化学习实现反应式多智能体编排

## 引言（草稿 — 目标：AAAI 2027）

大语言模型在问答、数学推理和代码生成等任务上展现了卓越的能力。然而，没有任何单一模型能在所有任务或同一任务的所有难度级别上都表现最优。这一观察催生了大量关于 **LLM 路由与多智能体编排** 的研究——通过协调多个模型来实现更高性能、更低成本，或兼而有之。

现有的多模型协调方法可归纳为三种范式，每种都存在根本性的局限。

**单轮路由器**（RouteLLM [Ong et al., 2025]；RouterDC [Chen et al., 2024]；GraphRouter [Feng et al., 2025]；Eagle [Lu et al., 2024]）根据预测的难度或成本-质量权衡，为每条查询选择一个模型。这类方法对简单查询有效，但做出的是不可撤回的一次性决策：如果所选模型失败或查询需要迭代推理，则没有恢复、分解或精炼的机制。Router-R1（Zhang et al., 2025）将这一范式扩展到多轮，通过 PPO 训练 LLM 发出 `<search>Model:query</search>` 调用，但其路由目标是仅在能力和成本上有所区别的同质 LLM——它无法分解任务、验证中间结果或跨子答案进行综合。

**开环编排器**（Conductor [Nielsen et al., 2026]；AgentConductor [Wang et al., 2026]；FlowReasoner [Guo et al., 2025]）代表了一个重要进步：它们使用强化学习训练一个小模型来设计多智能体工作流。Conductor 学习为一组前沿模型生成通信拓扑和定制化的 worker 提示词，在 GPQA 和 LiveCodeBench 上取得了优异成绩。AgentConductor 进一步引入了密度感知的 DAG 生成用于代码任务。然而，这些系统共享一个关键的结构性局限：**编排计划在执行前生成，且在整个执行过程中保持不变**。编排器在未观察中间智能体输出的情况下，预先确定拓扑或工作流。当某个 worker 返回低质量或矛盾的信息时——这在多样化的模型池中很常见——系统无法自适应调整、用不同的智能体重试或重新构造子查询。这种开环设计也使得显式成本约束难以融入，因为总资源消耗由预先确定的计划决定，而非由观察到的实际需求决定。

**流水线精炼**方法（HierRouter [Li et al., 2025]；EvoRoute [Xu et al., 2026]）从另一个角度切入：它们将输出依次传递给一系列模型，每一步精炼上一步的答案。HierRouter 训练一个基于 PPO 的 RL 智能体来动态选择由哪个专业模型处理每个精炼步骤，将多跳推理建模为 MDP。EvoRoute 维护一个经验知识库，基于性能、成本和延迟的帕累托最优进行选择。虽然这些方法实现了多步协调，但其路由器是一个**纯粹的调度器**——它决定*谁*来处理下一步，但不决定*问什么*。完整上下文被原样转发，没有任务分解、定向子查询或跨智能体验证。这限制了它们在需要组合推理或多源信息综合的任务上的有效性。

我们识别出三个上述范式均未完全解决的开放挑战：

1. **反应式闭环编排。** 编排器应在观察每个智能体的响应后再决定下一步动作——从而实现错误恢复、自适应重路由，以及在收集到足够信息时提前终止。

2. **功能异构的智能体池。** 编排器不应仅路由到在能力/成本上有所不同的模型（Router-R1）或通信拓扑上有所不同的模型（Conductor），而应协调具有不同认知角色的智能体——分解、执行、验证、综合——并学会何时需要每种角色。

3. **成本条件化的自适应行为。** 系统不应仅将成本作为惩罚项来最小化，而应在不同成本预算下展现出质的不同的策略——对简单查询使用更少、更便宜的智能体，仅在任务确实需要时才投入多轮验证。

我们提出 **OrchestratorR1**，一个在统一的强化学习框架内同时解决上述三个挑战的系统。我们将多智能体编排建模为马尔可夫决策过程（MDP），其中一个小型语言模型（Qwen2.5-3B/7B）作为编排器策略 $\pi_\theta$。在每一步，编排器观察完整的交互历史——包括所有先前的智能体响应——然后决定是用一个**定制化的子查询**调用特定智能体，还是输出最终答案。智能体池由四个功能专业化的角色组成：**executor**（任务执行，支持 strong/weak 模型档位）、**decomposer**（多跳任务分解）、**critic**（中间结果验证）和 **synthesizer**（跨答案聚合）。每个角色由一个来自 strong worker pool（Claude Sonnet 4 / Gemini 2.5 Pro / GPT-4o）的固定 LLM API 端点支撑。关键在于，编排器是唯一被训练的组件；智能体池是冻结的基础设施。我们明确指出：四个角色定义是编码了不同认知功能的人工先验，但**何时调用每个角色、以什么顺序、用什么子查询、是否调用 critic 进行验证，都完全由 RL 奖励景观涌现**。

为了在有限的上下文窗口内支持多轮编排，我们采用上下文压缩机制：当上下文超出预算时，保留任务描述和最近的交互，对早期的智能体响应进行摘要压缩。

训练分三个阶段进行：（1）简短的 SFT 热身，教会模型 `<call type="...">query</call>` 输出格式；（2）渐进式 GRPO（Group Relative Policy Optimization）主训练，逐步增加编排 horizon——从 max_turns=2 的简单 QA 开始，扩展到 max_turns=4 的多跳数据，最终在 max_turns=6 的全量混合数据上训练。这种渐进式调度借鉴了长 horizon 智能体训练中的课程学习策略（KLong; Xiong et al., 2025），让编排器先掌握基本路由，再学习复杂的多轮工作流。复合奖励为 $R(\tau) = R_\text{outcome} - \alpha \cdot C_\text{cost} - \beta \cdot C_\text{turns} + \gamma \cdot B_\text{efficiency}$。在这一单一目标下，编排器自发涌现出**自适应行为谱**：简单事实查询通过一次 executor 调用即可解决；多跳推理触发任务分解、并行执行和综合；复杂任务还会额外调用 critic 进行中间验证。这些策略不是人工设计的规则，而是由 RL 奖励景观塑造的涌现行为。

我们的贡献如下：

1. **将反应式编排建模为 MDP。** 我们首次将多智能体编排形式化为闭环序贯决策过程，编排器基于观察到的智能体响应来决定每一步动作。这使得错误恢复、自适应重路由和提前终止成为可能——这些能力在开环规划器（Conductor、FlowReasoner）和流水线精炼器（HierRouter）中均不具备。

2. **功能异构的智能体协调与有原则的设计。** 我们协调四个具有不同认知角色的智能体（executor / decomposer / critic / synthesizer），并明确区分什么是设计、什么是学习：角色定义是编码认知功能的人工先验，而调用时机、顺序、子查询表述和验证决策都由 RL 涌现。系统性消融实验（移除每个角色、加回第五个角色）验证了四角色设计的必要性和充分性。

3. **涌现的自适应策略与成本-质量帕累托优化。** 我们证明单一 RL 目标能在不同任务复杂度下产生质的不同的编排策略，且成本系数 $\alpha$ 通过调节固定 worker pool 内的调用频率和轮数，提供了对成本-质量帕累托前沿的显式控制。

4. **跨模态泛化。** 单一训练策略同时处理问答、多跳推理和代码生成，覆盖 10 个基准测试，而此前的 RL 训练编排器仅专注于一到两种任务类型（Conductor：推理+代码；Router-R1：仅问答；AgentConductor：仅代码）。

我们在三个赛道上评估 OrchestratorR1：简单问答（NQ、TriviaQA、PopQA）、多跳推理（HotpotQA、2WikiMultihopQA、MuSiQue、Bamboogle）和代码生成（HumanEval、MBPP）。在匹配 worker 条件下，我们与 Conductor 在共有基准（GPQA、LiveCodeBench）上进行控制变量对比，将编排范式的效果与 worker 能力隔离。我们的反应式范式取得了可比的结果，同时通过闭环错误恢复展示了结构性优势。在共有基准之外，OrchestratorR1 泛化到了 Conductor 未评估的问答和多跳推理任务。消融实验证实，反应式闭环机制、critic 智能体和 decomposer 各自贡献显著，其中仅反应式设计就在多跳任务上贡献了 X% 的性能提升。
