# OrchestratorR1: Learning Reactive Multi-Agent Orchestration via Reinforcement Learning

## Introduction (Draft — Target: AAAI 2027)

Large language models have demonstrated remarkable capabilities across question answering, mathematical reasoning, and code generation. However, no single model excels at all tasks or all difficulty levels within a task. This observation has motivated a growing body of work on **LLM routing and multi-agent orchestration** — systems that coordinate multiple models to achieve better performance, lower cost, or both.

Existing approaches to multi-model coordination fall into three paradigms, each with fundamental limitations.

**Single-round routers** (RouteLLM [Ong et al., 2025]; RouterDC [Chen et al., 2024]; GraphRouter [Feng et al., 2025]; Eagle [Lu et al., 2024]) select one model per query based on predicted difficulty or cost-quality tradeoffs. While effective for simple queries, they make an irrevocable one-shot decision: if the chosen model fails or the query requires iterative reasoning, there is no mechanism for recovery, decomposition, or refinement. Router-R1 (Zhang et al., 2025) extends this paradigm to multiple rounds by training an LLM via PPO to emit `<search>Model:query</search>` calls, but its routing targets are homogeneous LLMs distinguished only by capability and cost — it cannot decompose tasks, verify intermediate results, or synthesize across sub-answers.

**Open-loop orchestrators** (Conductor [Nielsen et al., 2026]; AgentConductor [Wang et al., 2026]; FlowReasoner [Guo et al., 2025]) represent a significant advance: they use reinforcement learning to train a small model that designs multi-agent workflows. Conductor learns to generate communication topologies and per-worker prompts for a pool of frontier models, achieving strong results on GPQA and LiveCodeBench. AgentConductor extends this with density-aware DAG generation for code tasks. However, these systems share a critical structural limitation: **the orchestration plan is generated before execution and remains fixed throughout**. The orchestrator commits to a topology or workflow upfront, without observing intermediate agent outputs. When a worker returns low-quality or contradictory information — a common occurrence with diverse model pools — the system cannot adapt, retry with a different agent, or reformulate the sub-query. This open-loop design also makes it difficult to incorporate explicit cost constraints, as the total resource consumption is determined by the pre-committed plan rather than by observed necessity.

**Pipeline refinement** approaches (HierRouter [Li et al., 2025]; EvoRoute [Xu et al., 2026]) take a different angle: they sequentially pass outputs through a chain of models, with each step refining the previous answer. HierRouter trains a PPO-based RL agent to dynamically select which specialized model handles each refinement step, modeling multi-hop reasoning as an MDP. EvoRoute maintains an experience knowledge base and uses Pareto-optimal selection across performance, cost, and latency. While these methods achieve multi-step coordination, their router is a **pure dispatcher** — it decides *who* processes next but not *what* to ask. The full context is forwarded as-is, without task decomposition, targeted sub-queries, or cross-agent verification. This limits their effectiveness on tasks requiring compositional reasoning or information synthesis from multiple sources.

We identify three open challenges that none of the above paradigms fully address:

1. **Reactive closed-loop orchestration.** The orchestrator should observe each agent's response before deciding the next action — enabling error recovery, adaptive re-routing, and early termination when sufficient information has been gathered.

2. **Functionally heterogeneous agent pools.** Rather than routing to models that differ only in capability/cost (Router-R1) or communication topology (Conductor), the orchestrator should coordinate agents with distinct cognitive roles — decomposition, execution, verification, synthesis — and learn when each role is needed.

3. **Cost-conditioned adaptive behavior.** The system should not merely minimize cost as a penalty term, but exhibit qualitatively different strategies under different cost budgets — using fewer, cheaper agents for simple queries and investing in multi-round verification only when the task demands it.

We propose **OrchestratorR1**, a framework that addresses all three challenges within a unified reinforcement learning formulation. We model multi-agent orchestration as a Markov Decision Process where a small language model (Qwen2.5-3B/7B) serves as the orchestrator policy $\pi_\theta$. At each step, the orchestrator observes the full interaction history — including all prior agent responses — and decides either to invoke a specific agent with a **custom-crafted sub-query**, or to emit a final answer. The agent pool consists of four functionally specialized roles: **executor** (task execution, supporting strong/weak model tiers), **decomposer** (multi-hop task decomposition), **critic** (intermediate result verification), and **synthesizer** (cross-answer aggregation). Each role is backed by a fixed LLM API endpoint from a strong worker pool (Claude Sonnet 4 / Gemini 2.5 Pro / GPT-4o). Crucially, the orchestrator is the only trained component; the agent pool is frozen infrastructure. We note that the four role definitions are human priors encoding distinct cognitive functions, but **when to invoke each role, in what order, with what sub-query, and whether to invoke the critic at all are entirely emergent from the RL reward landscape**.

To support multi-round orchestration within bounded context windows, we employ a context compression mechanism that preserves the task description and recent interactions while summarizing earlier agent responses when the context budget is exceeded.

Training proceeds in three stages: (1) a brief SFT warmup that teaches the `<call type="...">query</call>` output format, followed by (2) progressive GRPO (Group Relative Policy Optimization) that gradually increases the orchestration horizon — starting with max_turns=2 on simple QA, expanding to max_turns=4 with multi-hop data, and finally max_turns=6 on the full mixture. This progressive schedule, inspired by curriculum strategies in long-horizon agent training (KLong; Xiong et al., 2025), allows the orchestrator to first master basic routing before learning complex multi-round workflows. The composite reward is $R(\tau) = R_\text{outcome} - \alpha \cdot C_\text{cost} - \beta \cdot C_\text{turns} + \gamma \cdot B_\text{efficiency}$. Under this single objective, the orchestrator spontaneously develops an **adaptive behavior spectrum**: simple factual queries are resolved in one call to the executor; multi-hop reasoning triggers decomposition into sub-tasks with parallel execution and synthesis; complex tasks additionally invoke a critic for intermediate verification. These strategies are not hand-designed rules but emergent behaviors shaped by the RL reward landscape.

Our contributions are as follows:

1. **Reactive orchestration as MDP.** We are the first to formulate multi-agent orchestration as a closed-loop sequential decision process where the orchestrator conditions each action on observed agent responses. This enables error recovery, adaptive re-routing, and early stopping — capabilities absent in open-loop planners (Conductor, FlowReasoner) and pipeline refiners (HierRouter).

2. **Functionally heterogeneous agent coordination with principled design.** We coordinate four agents with distinct cognitive roles (executor / decomposer / critic / synthesizer) and explicitly distinguish what is designed from what is learned: role definitions are human priors encoding cognitive functions, while invocation timing, ordering, sub-query formulation, and verification decisions are all emergent from RL. Systematic ablations (removing each role, adding back a fifth) validate the necessity and sufficiency of the four-role design.

3. **Emergent adaptive strategies with cost-quality Pareto optimization.** We demonstrate that a single RL objective produces qualitatively different orchestration strategies across task complexities, and that the cost coefficient $\alpha$ provides explicit control over the cost-quality Pareto frontier by modulating invocation frequency and turn count within a fixed worker pool.

4. **Cross-modal generalization.** A single trained policy handles QA, multi-hop reasoning, and code generation across 10 benchmarks, whereas prior RL-trained orchestrators specialize in one or two task types (Conductor: reasoning + code; Router-R1: QA only; AgentConductor: code only).

We evaluate OrchestratorR1 on three tracks: simple QA (NQ, TriviaQA, PopQA), multi-hop reasoning (HotpotQA, 2WikiMultihopQA, MuSiQue, Bamboogle), and code generation (HumanEval, MBPP). Under matched worker conditions, we conduct a controlled comparison with Conductor on shared benchmarks (GPQA, LiveCodeBench), isolating the effect of orchestration paradigm from worker capability. Our reactive paradigm achieves comparable results while demonstrating structural advantages through closed-loop error recovery. Beyond shared benchmarks, OrchestratorR1 generalizes to QA and multi-hop reasoning tasks where Conductor was not evaluated. Ablation studies confirm that the reactive closed-loop mechanism, the critic agent, and the decomposer each contribute significantly, with the reactive design alone accounting for X% of the performance gain on multi-hop tasks.
