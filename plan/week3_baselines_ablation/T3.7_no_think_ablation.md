## 为什么新增这个消融

`<think>` 标签在当前系统中是一个**无监督的 scratchpad**：
- `parser.py` 只检查 `<think>` 是否存在（`has_think = True/False`），不读取内容
- `reward.py` 完全不关心 think 内容——reward 只看 `<call>` 和 `<answer>` 的结果
- 模型在 think 里写什么对系统没有任何直接影响

审稿人几乎一定会问："你怎么知道 `<think>` 不是冗余的 filler？" 如果删掉 think 性能不变，那 think 就是噪声。

## 实验设计

### 修改内容
1. **system_prompt.py**: 移除 `<think>` 相关描述和示例
   - 删除 "Always start with `<think>` to reason about the task complexity"
   - 删除示例中的 `<think>` 块
   - 让模型直接输出 `<call>` 或 `<answer>`

2. **SFT 数据**: 生成 ~50 条无 think 的 SFT 示例
   - 从现有 SFT 数据中批量删除 `<think>...</think>` 块即可
   
3. **parser.py**: 不需要修改（has_think 字段只用于统计，不影响逻辑）

4. **reward.py**: 不需要修改（validate_format 不要求 think 存在）

### 训练配置
- 基于同一基座模型
- 同样的 GRPO 超参数
- **唯一区别**: system prompt 中无 think 指令，SFT 数据中无 think 示例

## 运行

```bash
# 1. 生成无 think 的 SFT 数据
python data_process/strip_think.py \
    --input data/sft_warmup.jsonl \
    --output data/sft_warmup_no_think.jsonl

# 2. SFT（无 think 格式）
accelerate launch --config_file training/accelerate_fsdp_a100_4gpu.yaml \
    training/sft_warmup.py \
    --model_path models/Qwen2.5-7B-Instruct \
    --data_path data/sft_warmup_no_think.jsonl \
    --output_dir checkpoints/sft_no_think_7b \
    --no_think

# 3. GRPO（无 think）
bash training/train_a100.sh \
    --model_path checkpoints/sft_no_think_7b \
    --output_dir checkpoints/ablation_no_think \
    --no_think \
    --seed 42

# 4. 评估
python eval/eval_orchestrator.py \
    --model_path checkpoints/ablation_no_think/final \
    --data_path data/test_qa.jsonl \
    --measure_latency \
    --output eval/results/ablation_no_think.json
```

## 配合分析（Section 5.3）

无论消融结果如何，都需要对 think 内容做事后分析（在有 think 的主模型上）：
1. Think 长度 vs. 题目复杂度分布
2. Think 内容与后续行为的一致性
3. Think 中的关键动作词频率
