# T3.2 w/o Reactive 消融 ⚠️ 最关键实验

## 基本信息
- **优先级**: **P0 最高**
- **预计耗时**: 训练 8-12h + 评估 4h
- **依赖**: W2 SFT checkpoint
- **产出**: `eval/results/orch_no_reactive_7b.json`, `checkpoints/orch_no_reactive_7b/`
- **状态**: [ ] 未开始

## 目标
实现 "开环规划" 版本的 OrchestratorR1（模拟 Conductor 模式），对比 "闭环反应式"。
**这是回答 "为什么不用 Conductor" 的核心证据。**

## 什么是 "w/o reactive"

### 正常模式 (Reactive / 闭环):
```
模型: <think>分析...</think><call type="decomposer">分解任务</call>
系统: <information>分解结果: 1. xxx 2. yyy</information>     ← 模型能看到
模型: <call type="executor_cheap">查询 xxx</call>             ← 基于上一步结果决策
系统: <information>xxx 的答案是 zzz</information>              ← 模型能看到
模型: <call type="synthesizer">综合...</call>                  ← 再次基于结果决策
...
```

### w/o Reactive (开环 / 一次性规划):
```
模型: <think>分析...</think>
      <call type="decomposer">分解任务</call>
      <call type="executor_cheap">查询 xxx</call>
      <call type="synthesizer">综合 xxx 和 yyy</call>
      <answer>最终答案</answer>
↑ 一次性输出所有步骤，不看中间结果
```
系统在模型输出全部 call 后，按顺序执行 agent 调用，最终用最后的 synthesizer 结果替换 answer。

## 实现方案

### 方案 A: 修改 generation.py 的 rollout 逻辑
```python
class NonReactiveGenerationManager:
    def rollout(self, user_input):
        # 1. 让模型一次性输出所有 <call> 和 <answer>
        # 2. 解析出所有 call 序列
        # 3. 按顺序执行（但不注入 <information>）
        # 4. 用最后一个 agent 的返回作为 answer
```

### 方案 B: 修改训练，让 GRPO 优化开环策略
```python
# 训练时: 模型输出全部 call → 批量执行 → 用最终结果计算 reward
# 不在中间注入 <information>
```

**推荐方案 B**: 需要重新训练一个 GRPO 模型。这样对比更公平——同样的训练方法，区别仅在于是否有中间反馈。

### 训练配置
- 基于同一 SFT checkpoint
- 同样的 GRPO 超参数（α, β, γ, G, lr, steps）
- **唯一区别**: generation loop 中不注入 `<information>`，模型一次性生成全部输出

## SFT 数据适配

需要额外生成 ~50 条"开环"SFT 示例：
```jsonl
{
  "input": "Who directed the film that won Best Picture when the Berlin Wall fell?",
  "output": "<think>This is a multi-hop question. I'll plan the full sequence upfront.</think>\n<call type=\"decomposer\">Break down: Who directed...</call>\n<call type=\"executor_cheap\">When did the Berlin Wall fall? What film won Best Picture that year?</call>\n<call type=\"executor_cheap\">Who directed that film?</call>\n<call type=\"synthesizer\">Combine the answers</call>\n<answer>The director was...</answer>"
}
```

## 预期结果与论点

| 指标 | Reactive (Ours) | Non-Reactive | 差异来源 |
|------|-----------------|--------------|---------|
| F1 (简单QA) | ~55% | ~50% | 简单题差异小 |
| F1 (多跳推理) | ~50% | ~35% | ⚠️ 大幅差异! 多跳题需要看中间结果 |
| F1 (复杂代码) | ~45% | ~30% | ⚠️ 需要看执行反馈 |
| Format rate | >90% | >85% | 开环格式更简单但质量更不可控 |
| 平均成本 | $0.004 | $0.008 | 开环不能提前终止，总是执行全部 call |

**关键论点**: 多跳推理和复杂任务上，reactive 模式显著优于 non-reactive，因为后续 agent 调用可以根据前一步返回的信息做出更精准的 query。
