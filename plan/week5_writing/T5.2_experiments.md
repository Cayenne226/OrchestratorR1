## 结构 (~3.5 页)

### 4.1 Experimental Setup (~0.7 页)

**Datasets**:
- Track 1 (Simple QA): NQ, TriviaQA, PopQA (500 test each)
- Track 2 (Multi-hop): HotpotQA, 2WikiMultihop, MuSiQue (500 each)
- Track 3 (Code): HumanEval (164), MBPP (427)
- Track 4: GPQA Diamond (198), LiveCodeBench (202)

**Baselines**:
| Baseline | 一句话描述 |
|----------|-----------|
| Direct-Strong | Single strong LLM, no orchestration |
| Direct-Cheap | Single cheap LLM, no orchestration |
| Router-R1 | RL-learned routing to homogeneous LLMs (NeurIPS'25) |
| Conductor | RL-learned open-loop workflow generation (ICLR'26) |
| ReAct | Zero-shot tool-use agent (Qwen2.5-7B) |
| Fixed-Pipeline | All 6 agents in fixed order |
| Self-Reflection-5 | 5-round self-critique with same model |

**Metrics**: EM, F1 (QA); Pass@1 (Code); Accuracy (GPQA); **Avg Cost ($); Avg Turns; Avg Latency (s)**

> **新增 Avg Latency**: 每个 method 报告平均每条 query 的 wall-clock 延迟（含模型推理 + API 调用）。这是对 reactive 串行代价的诚实报告。

**Implementation**: Qwen2.5-7B-Instruct, GRPO G=16, 4×A100 80GB, ~8-12h training

### 4.2 Reactive vs. Open-Loop: Robustness Under Failure (~0.8 页) ⚠️ 核心

**Table 1: Adversarial Robustness (Contribution #1 的核心证据)**

> 对应论文 Contribution #1: "Closed-loop feedback is critical for robust multi-agent orchestration"

```
| noise_rate | Reactive F1 | Open-Loop F1 | Δ F1   | Reactive Latency | Open-Loop Latency |
|------------|-------------|--------------|--------|------------------|-------------------|
| 0.0        | X ± σ       | Y ± σ        | +Z     | A.As             | B.Bs              |
| 0.1        |             |              |        |                  |                   |
| 0.2        |             |              |        |                  |                   |
| 0.3        |             |              |        |                  |                   |
| 0.5        |             |              |        |                  |                   |
```

**配套 Figure 5**: F1 vs. noise_rate 折线图（同一图中）

**关键叙述**:
1. noise=0 时，reactive 已有优势（中间反馈改善多跳推理）
2. 随 noise 增大，差距显著扩大（reactive 能检测并恢复错误）
3. open-loop 在 noise=0.3 时性能崩溃（error cascading）
4. **诚实承认**: reactive 延迟随 noise 增加（更多重试轮次），这是 robustness 的 trade-off
5. "Reactive trades latency for robustness — a controlled trade-off that open-loop cannot make"

**Case study**: 紧跟 Table 1 后，用 1/4 页展示一个具体的 reactive recovery 实例

### 4.4 Matched-Worker Comparison (~0.5 页)

> 对应论文 Contribution #3: "Orchestration strategies transfer zero-shot"

```
| Method             | GPQA  | LCB   | Worker Pool              | Paradigm  |
|--------------------|-------|-------|--------------------------|-----------|
| Conductor†         | 87.5  | 83.93 | GPT-5,Claude,Gemini,...  | Open-loop |
| Self-Reflect-5     | TBD   | TBD   | GPT-5                   | Self-refl |
| Direct-GPT-5       | TBD   | TBD   | GPT-5                   | Direct    |
| Ours (matched)     | TBD   | TBD   | GPT-5,Claude,Gemini,...  | Reactive  |
| Ours (cheap)       | TBD   | TBD   | GPT-4o-mini,GPT-4o,...   | Reactive  |

```

**关键叙述**:
1. 同等 worker 下，reactive ≥ open-loop (Conductor)
2. Ours (cheap) vs. Ours (matched)：同一 checkpoint，策略迁移有效
3. **成本对比要诚实**：美元成本差异主要来自 worker pool 选择（10-50x），不来自执行范式
4. 执行范式的美元成本差异较小：reactive 简单题省钱，复杂题成本相当
