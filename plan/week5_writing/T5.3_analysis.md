### 5.2 Robustness-Latency Trade-off (~0.6 页)

> ⚠️ 此节不再是简单的 "reactive vs open-loop" 展示，而是诚实分析 **robustness 和 latency 的 trade-off**。

- **核心数据来自 Table 1 (Adversarial)**，此节做深入分析
- **Figure 5**: F1 vs noise_rate（如果 Table 1 中已有折线图，此处做补充分析）

**分析要点**:

1. **Error cascading 现象量化**:
   - Open-loop 下，一个错误的中间结果如何传播到后续步骤
   - 统计：在 noise=0.3 时，open-loop 中 X% 的最终错误可追溯到第一个错误的 agent 返回

2. **Reactive 恢复机制分类**:
   - 统计恢复路径的分布：多少次是 critic→retry，多少次是换 agent，多少次是 refine query
   - "The model learns multiple recovery strategies, not a single pattern"

3. **延迟 trade-off 的诚实分析**:
   - noise=0 时：reactive 平均延迟 vs. open-loop，差异有多大
   - noise=0.3 时：reactive 延迟增加（因为重试），但这是"用时间换正确性"
   - 混合工作负载（60% 简单 + 40% 复杂）下的加权平均延迟
   - **关键论点**: "Reactive trades latency for robustness — a trade-off that open-loop cannot make at any latency budget"

4. **Case study**: 1 个 reactive recovery 实例（简化版，从 T4.4 选取）

### 5.3 What Does `<think>` Actually Learn? (~0.5 页) ⚠️ 新增

> **新增原因**: `<think>` 的内容对系统完全无影响（parser 不读取 think 内容，reward 不评估 think 质量）。审稿人会问："think 真的在工作，还是只是输出噪声？" 此节用事后分析（不改方法）提供证据。

**分析内容**:

1. **w/o think 消融结果**（数据来自 Table 4）:
   - 如果 w/o think 性能下降 → think 不是冗余的
   - 报告下降幅度和在哪些任务类型上下降最多

2. **Think 长度 vs. 题目复杂度分布** (Figure X):
   - 统计所有 rollout 的 think 字符长度
   - 按 difficulty 分组 (simple / multi_hop / code)
   - **预期**: 多跳题的 think 显著更长
   - 如果成立 → 模型涌现出了复杂度感知的 reasoning

3. **Think 内容与后续行为一致性**:
   - 如果 think 中出现 "simple" / "directly" / "straightforward" → 后续行为是否真的是 1 步 executor_cheap
   - 如果 think 中出现 "decompose" / "complex" / "multiple steps" → 后续行为是否真的调用了 decomposer
   - 统计一致率

4. **Think 中的关键动作词频率** (Figure X):
   - 简单题 think 中的高频词 vs. 多跳题 think 中的高频词
   - Word cloud 或 bar chart

**核心论点**:
> "Although `<think>` content receives no direct supervision — the reward signal is solely determined by the outcomes of `<call>` and `<answer>` actions — post-hoc analysis reveals that the model spontaneously develops meaningful, complexity-aware reasoning. Think blocks for multi-hop questions are X% longer on average and contain significantly more planning-related vocabulary."

**在论文 Method 中的配套声明**:
> "Unlike approaches that require explicit task difficulty labels, our orchestrator learns to assess task complexity implicitly through the RL reward signal. The `<think>` tag serves as an unconstrained reasoning space — the model receives no supervision on what to think, only on the outcomes of its actions."
