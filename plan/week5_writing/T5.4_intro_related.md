## Introduction 结构 (~1.5 页)

### Para 1: 背景 + 痛点
LLM 已经很强大，但单一 LLM 有局限 → 多 Agent 系统。
但现有多 Agent 框架依赖人工设计（AutoGen, CrewAI）→ 能否端到端学习编排策略？

### Para 2: 现有工作的进展与局限
- Router-R1: 证明"路由"可以 RL 学习，但仅选模型、无功能分工
- Conductor: 证明"编排"可以 RL 学习，但一次性生成完整 workflow（开环），无法中间调整
- AgentConductor: DAG 拓扑优化，但限于代码任务
- **关键定位**: Conductor 回答了 "能不能学编排"，我们回答 "编排时 closed-loop feedback 到底有多重要"

### Para 3: 本文方法
OrchestratorR1: 反应式多 Agent 编排
- Orchestrator 逐步观察 agent 返回、逐步决策（closed-loop control）
- 6 种功能角色（refiner, decomposer, executor, critic, synthesizer）
- GRPO 端到端训练，单一复合奖励函数
- 涌现自适应行为：简单题 1 步终止，复杂题多步分解+验证

### Para 4: 核心发现
- **核心实验**: Adversarial intermediate failure 实验定量证明 reactive 的鲁棒性优势（在 30% 错误率下，reactive 保持 X% 性能 vs. open-loop Y%）
- **自适应效率**: 混合工作负载下 reactive 平均延迟反而更低（简单题 1 步终止的收益 > 复杂题串行的代价）
- 一个策略跨 QA/推理/代码三种模态泛化
- 编排策略从 cheap worker 零样本迁移到 frontier worker

### Para 5: Contributions (金字塔结构)

**核心 Claim**:
> Closed-loop reactive orchestration significantly outperforms open-loop planning for multi-agent systems, especially under intermediate failures.

**三个 Contributions**:

1. **We show that closed-loop feedback is critical for robust multi-agent orchestration.** Through controlled experiments (same model, same agent pool, only changing the execution paradigm) and adversarial evaluation (injecting agent errors at rates 10%-50%), we demonstrate that reactive orchestration degrades gracefully while open-loop planning suffers catastrophic error cascading. (Section 5.2, Table 3, Figure 5)

2. **We demonstrate that a single reactive policy generalizes across QA, multi-hop reasoning, and code tasks**, covering 10 benchmarks that no prior system addresses jointly. The policy adaptively emerges different strategies per task type: 1-step shortcuts for simple factual queries, multi-step decompose-verify chains for multi-hop reasoning. (Section 4.2, Table 2, Figure 2)

3. **We show that orchestration strategies transfer zero-shot from cost-efficient workers to frontier models**, enabling explicit cost-quality Pareto optimization via a single hyperparameter α. (Section 4.2 Table 1 vs Table 2, Section 4.4 Figure 4)

## Related Work 结构 (~1 页)

### LLM Routing and Selection
- RouteLLM, FrugalGPT, AutoMix
- **Router-R1**: 最相关前作，RL 学路由
- 局限: 同质模型选择，无功能分工

### RL-Learned Agent Orchestration
- **Conductor** (ICLR'26): GRPO 训练，开环 workflow 生成，推理 SOTA
  - 关键区别: open-loop vs. closed-loop (cite 控制论经典)
  - 公正评价: Conductor 的通信拓扑和并行执行是结构性创新
  - 我们的补充: 回答了 "intermediate feedback 何时必要" 这个 Conductor 未回答的问题
- **AgentConductor**: SFT+GRPO，DAG 拓扑，代码任务
- 与本文区别: reactive vs open-loop, 功能角色 vs 模型选择, 跨模态

### Multi-Agent Systems
- AutoGen, CrewAI, LangGraph: 手工设计
- CAMEL, MetaGPT: 角色扮演
- 与本文区别: 端到端 RL 学习 vs 人工设计

### Agentic Reinforcement Learning
- GRPO (DeepSeek-R1), iStar, IGPO
- Tool-use RL: ToolFormer, Agent-R1
- 本文贡献: 多 agent 编排（非单 agent tool-use）

### Closed-Loop vs Open-Loop Control (新增)
- 控制论经典文献: closed-loop control 的抗扰动理论基础
- 在 LLM agent 系统中的对应: 
  - Open-loop = plan-then-execute (Conductor, AgentConductor)
  - Closed-loop = observe-decide-act (OrchestratorR1, Router-R1)
- 我们首次在多 agent orchestration 场景下定量比较两种范式
