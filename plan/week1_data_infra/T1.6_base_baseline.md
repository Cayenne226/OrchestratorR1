# T1.6 Base Model Baseline

## 基本信息
- **优先级**: P1
- **预计耗时**: 0.5 天
- **依赖**: T1.1-T1.3 (数据), T1.5 (环境)
- **产出**: `eval/results/orch_base_7b.json`
- **预计 API 成本**: ~$5-10
- **状态**: [ ] 未开始

## 目标
评估**未训练**的 Qwen2.5-7B-Instruct 在 Orchestrator 框架下的零样本表现，作为 baseline 下界。

## 为什么需要这个
- 证明 SFT + GRPO 训练的增益不是 base model 自带的
- 预期: base model 不会正确使用 `<call>` 标签，大多数输出格式错误
- 如果 base model 已经表现不错 → 说明框架设计好但 RL 贡献小（需要调整 story）

## 实现步骤

### Step 1: 运行评估
```bash
python eval/eval_orchestrator.py \
    --model_path models/Qwen2.5-7B-Instruct \
    --data_path data/test_qa.jsonl \
    --api_base YOUR_API_BASE \
    --api_key YOUR_API_KEY \
    --output eval/results/orch_base_7b_qa.json \
    --max_samples 500

python eval/eval_orchestrator.py \
    --model_path models/Qwen2.5-7B-Instruct \
    --data_path data/test_code.jsonl \
    --api_base YOUR_API_BASE \
    --api_key YOUR_API_KEY \
    --output eval/results/orch_base_7b_code.json

# GPQA 和 LiveCodeBench 如果评估脚本就绪也跑
```

### Step 2: 记录关键指标
- Format valid rate (多少比例的输出是合法的 `<call>`/`<answer>` 格式)
- EM / F1 (QA)
- Pass@1 (代码)
- 平均轮数
- 平均成本
- Agent type 分布

### Step 3: 合并结果
```bash
python eval/compare.py \
    --results eval/results/orch_base_7b_qa.json eval/results/orch_base_7b_code.json \
    --output eval/results/orch_base_7b.json
```

## 预期结果

| 指标 | 预期值 | 说明 |
|------|--------|------|
| Format valid rate | 10-30% | 未训练模型大概率不会生成正确的 `<call>` 标签 |
| EM (简单 QA) | 5-15% | 大多数格式错误导致无法提取答案 |
| F1 (多跳推理) | 5-10% | 同上 |
| Pass@1 (代码) | 0-5% | 代码任务更难 |
| 平均轮数 | 1.0-1.5 | 格式错误导致提前终止 |
